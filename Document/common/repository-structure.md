# 仓库结构说明

## 概述

本仓库是 JustGo 的 monorepo，正式产物、工程代码、长期文档与 AI 协作过程文件同时存在。为了避免目录继续膨胀，需要明确每类目录的职责边界，减少“文件能放就放”的情况。

## 目录职责

```text
JustGo/
├── JustGo-backend/   # Spring Boot 后端工程
├── JustGo-frontend/  # Vue 3 前端工程
├── Document/         # 正式项目文档、模块文档、通用规范
├── docs/             # AI 协作过程产物（已 git ignore）
├── AGENTS.md         # 仓库级协作规范
├── CLAUDE.md         # 其他智能体兼容说明
└── README.md         # 项目入口说明
```

## 涉及文件

- `README.md`：项目简介、快速开始、仓库总览入口
- `AGENTS.md`：仓库级开发约束和文档规范
- `JustGo-backend/`：后端源码、资源文件、测试
- `JustGo-frontend/`：前端源码、静态资源、工程配置
- `Document/backend/`：后端模块正式文档
- `Document/common/`：通用机制、规范、避坑、仓库结构说明
- `Document/frontend/`：当前仅存放辅助性前端协作材料，不应继续扩张为正式产品文档区
- `docs/`：规划稿、临时 spec、AI superpowers 产物，仅供过程协作，不作为正式文档沉淀

## 当前结构问题

1. `Document/` 与 `docs/` 都在承载“文档”语义，但没有显式边界说明，新成员容易混淆。
2. 根目录缺少仓库地图，无法快速判断正式文档、临时产物、业务代码分别在哪里。
3. `.idea/`、`.vscode/`、`dist/` 等本地生成物没有在根忽略规则中统一收口，容易制造噪音。
4. `Document/frontend/` 与仓库规范“前端无需单独文档”存在轻微张力，后续若继续往里堆产品文档，会和当前规则冲突。

## 已做调整

1. 补充根目录 `.gitignore`，统一忽略 IDE 配置、前端构建产物、日志与常见本地缓存。
2. 在 `README.md` 增加仓库结构说明，降低上手成本。
3. 新增本文件，明确 `Document/` 与 `docs/` 的职责边界，避免后续继续混放。

## 结构治理建议

### 建议一：正式文档只进入 `Document/`

- 模块实现说明、避坑文档、通用组件说明统一沉淀到 `Document/`
- `docs/` 只保留 AI 规划稿、过程 spec、一次性讨论材料

### 建议二：为临时目录设置“只进不出”的约束

- `docs/` 继续保持 git ignore
- 需要长期保留的信息，要在功能完成后回写到 `Document/`

### 建议三：限制根目录继续长文件

- 新增工程目录前先判断是否属于前端、后端或正式文档
- 过程脚本、调试产物优先放子项目内，避免根目录成为杂物区

## 未解决的问题与隐患

### P1

- `Document/frontend/` 当前内容更像协作备忘录而非正式模块文档，后续若继续扩展，建议重新命名或迁移到统一的协作目录策略下。

### P2

- 仓库暂未提供统一的根级脚本入口（如一键启动前后端、统一检查命令），对于 monorepo 的开发体验仍有提升空间。

## 大厂标准对比

- 目录边界：当前已初步分层，但仍弱于成熟 monorepo 常见的 `apps/`、`services/`、`packages/`、`docs/` 清晰职责模型。
- 工程治理：正式文档与临时产物开始分流，但缺少自动化校验或目录约束。
- 开发体验：目前依赖人工记忆各子项目命令，尚未形成根级统一入口。

## 高可用 & 高并发分析

- 仓库结构本身不直接决定运行时高可用，但目录混乱会显著增加多人协作时的变更冲突与错误放置概率。
- 当项目规模扩大到更多服务或客户端时，建议升级为更显式的多应用结构，如 `apps/frontend`、`services/backend`、`packages/shared`，减少按历史命名堆叠的维护成本。
