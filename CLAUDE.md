# CLAUDE.md

此文件为 Claude Code（claude.ai/code）在本仓库中工作时提供指导。

## 项目概述

即行（JustGo）—— 一个发现城市活动、寻找搭子的平台(详细内容看README.md)。全栈 monorepo，包含 Vue 3 前端和 Spring Boot 后端，通过 `git subtree` 从独立仓库合并而成。

子目录 `JustGo-backend/CLAUDE.md` 和 `JustGo-frontend/CLAUDE.md` 分别包含后端和前端的架构细节。

## 开发命令

### 后端（`JustGo-backend/`）—— Spring Boot 4 + Java 17 + Maven

```bash
cd JustGo-backend
mvn spring-boot:run    # 启动开发服务器（端口 8080）
mvn test               # 运行测试
mvn clean package      # 构建 fat JAR
```

前置条件：MySQL 8（`localhost:3306/JustGo`，凭据见 `application.yaml`）和 Redis（`localhost:6379`）。首次启动前需执行 `src/main/resources/SQL/` 中的建表脚本。

### 前端（`JustGo-frontend/`）—— Vue 3 + Vite 8 + TypeScript 6

```bash
cd JustGo-frontend
npm install            # 安装依赖
npm run dev            # 启动开发服务器（端口 5173）
npm run build          # 类型检查 + 生产构建
npm run lint           # 运行 oxlint + eslint
npm run format         # Prettier 格式化
```

## 代码风格

Prettier 配置（`.prettierrc.json`）：无分号、单引号、每行最多 100 字符、2 空格缩进。

## 需求落地流程（企业级做法）

每个需求从接受到上线，必须走完以下流程。**不允许跳过任何步骤**。

### 阶段一：需求消化

- **用自己的话复述需求**：确认你理解的和产品经理想的一致。如果有歧义，先澄清再动手
- **拆解为子任务**：一个需求通常需要拆成 3-8 个子任务（数据模型 → API → 前端页面 → 联调 → 测试），每个子任务不超过半天工作量
- **标注依赖关系**：哪些子任务可以并行？哪些必须串行？API 设计完成后前后端可以并行开发
- **评估风险点**：涉及支付/认证/并发写/大数据量查询的，提前标注为高风险，优先设计方案

### 阶段二：技术方案设计

复杂需求（涉及新表/新模块/第三方集成）必须先写简版技术方案，存为 `docs/design/<模块名>.md`：

```
1. 背景：一句话说清为什么要做
2. 数据库变更：新表 DDL / 旧表加字段 / 索引变更
3. API 契约：端点签名 + request/response DTO 定义（前后端共同评审）
4. 核心流程：正常路径 + 异常路径的时序描述
5. 风险与对策：并发、数据一致性、第三方依赖
```

简单需求（单接口/单页面）直接在 PR 描述中写清楚 1-5 即可。

### 阶段三：分支与开发

```
master                    # 生产分支，只接受 PR 合入
  └─ feat/<功能名>        # 新功能分支，从 master 拉出
  └─ fix/<问题描述>       # 缺陷修复分支
```

- **分支命名**：全小写、短横线分隔，如 `feat/activity-signup`、`fix/follow-race-condition`
- **API 契约先行**：后端先定义 DTO 和端点签名并提交，前端基于契约 mock 数据并行开发
- **原子提交**：一个 commit 只做一件事（"加活动表 DDL"→"实现报名 API"→"加报名前端页面"），每个 commit 可独立回滚
- **WIP 提交用 `[WIP]` 前缀标记**，合入 master 前 squash 为干净的历史

### 阶段四：自测与验收

提交 PR 前必须完成：

- [ ] 正常路径走通（happy path 手动验证）
- [ ] 一个异常路径验证（输错参数/过期 token/重复提交）
- [ ] 并发场景脑测（两个请求同时到会怎样？）
- [ ] 缓存失效后功能正常（清掉 Redis 再试）
- [ ] 数据库回滚方案已准备好（涉及 DDL 变更的，写好回滚 SQL）
- [ ] 前端适配移动端（Chrome DevTools 切手机视图走一遍）

### 阶段五：PR 与审查

- PR 标题遵循 [Conventional Commits](https://www.conventionalcommits.org/)：`feat:` / `fix:` / `refactor:` / `docs:`
- PR 描述包含：做了什么 + 为什么这样做 + 如何验证（截图/curl 命令）
- 后端 PR 的强制检查项见 `JustGo-backend/CLAUDE.md#代码审查`
- **至少一人 approve 后方可合入**，不允许自己 approve 自己的 PR

### 阶段六：上线与观察

- 合入 master 后至少观察 10 分钟：错误日志、接口耗时、业务指标是否有异常波动
- 涉及 DDL 变更的，先上 DDL 再上代码（向后兼容），确保旧版本代码在新表结构上仍能运行
- 准备好回滚方案：代码回滚 + 数据回滚（如有），回滚比修复快的时候优先回滚

### 关键原则

- **简单需求不走火入魔**：修个 typo 不需要写方案文档，但涉及新表或新模块的必须写
- **坏消息第一时间暴露**：发现需求理解有误/技术方案有漏洞，立刻同步，不要自己悄悄修
- **API 变更是 breaking change**：改了 DTO 字段名或端点路径，必须同步通知前端，杜绝"我以为你知道"的假设
- **前向兼容**：数据库加字段必须有默认值或允许 NULL，旧代码不报错；删除字段分两次上线（先标记废弃，再物理删除）

## 模块文档规范

每完成一个功能模块，必须在 `Document/` 目录下输出对应的说明文档：
- 后端模块文档放在 `Document/backend/` 下
- 前端模块文档放在 `Document/frontend/` 下

文档需覆盖以下内容：

1. **概述**：模块职责与功能简介
2. **涉及文件**：列出所有相关文件的路径与用途
3. **API 端点**（如有）：HTTP 方法、路径、认证要求、说明
4. **实现细节**：核心流程的逐步描述、关键设计决策、数据结构
5. **已解决的问题**：该模块针对性地解决了哪些技术问题
6. **未解决的问题与隐患**：按严重程度（P0/P1/P2）分级列出
   - 安全漏洞（OWASP Top 10 相关）
   - 并发/竞态条件
   - 数据一致性风险
   - 死代码、逻辑不一致等代码质量问题
7. **大厂标准对比**：从高可用、高并发、安全性、可观测性等维度与行业主流方案对比，标注差距
8. **高可用 & 高并发分析**：当前瓶颈识别 + 不同量级（百万/千万/亿级）的规模化方案
9. **数据库**（如有）：表结构、关键字段、索引设计
